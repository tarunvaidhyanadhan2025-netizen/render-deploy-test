# RxLens

AI-powered prescription analysis platform. Upload a photo of any prescription and receive a structured, patient-friendly safety report for every medicine on it — in seconds.

> **Medical disclaimer:** RxLens is an informational tool only and does not constitute medical advice. Analysis results are generated by AI and may contain errors. Always consult your doctor, pharmacist, or qualified healthcare professional before making any changes to your medication.

## What is RxLens?

Prescription labels are dense. Generic names, brand names, dosages, and contraindications are written for clinicians — not patients. RxLens bridges that gap.

You upload an image of your prescription. RxLens:

1. **Reads it** — an OCR engine extracts all text from the image, including handwritten notes where legible.
2. **Finds the medicines** — an LLM refines the raw OCR output and identifies actual medicine names, filtering out noise.
3. **Analyses each medicine** — for every medicine found, the system generates a structured safety profile covering what the drug does, how to take it, common and serious side effects, drowsiness risk, age-specific warnings, and contraindications.
4. **Delivers a report** — the frontend renders the results as a clean, filterable safety report. Reports are cached locally for 24 hours and are printable.

The platform is provider-agnostic: it works with Google Gemini, OpenAI, locally-run Ollama models, HuggingFace transformers, or even a zero-API-key template fallback mode that uses pre-written YAML templates.

## How it Works

### System overview

```
┌─────────────────────────────────────────────────────────────┐
│                     Browser (Next.js)                        │
│                                                              │
│  /upload → drag image → OCR results → analysis progress      │
│  /results/:id → medicine cards, severity filter, print       │
│  /history → past prescriptions (backend + localStorage)      │
└──────────────────────────┬──────────────────────────────────┘
                           │ HTTP (JSON + multipart)
                           ▼
┌─────────────────────────────────────────────────────────────┐
│                    FastAPI Backend                            │
│                                                              │
│  POST /api/v1/upload                                         │
│    ├── Validate + save image                                 │
│    ├── OCR extraction (Tesseract / EasyOCR)                  │
│    ├── LLM name refinement                                   │
│    └── Save sidecar .meta file                               │
│                                                              │
│  POST /api/v1/analysis                                       │
│    ├── Load medicine list from .meta sidecar                 │
│    ├── For each medicine:                                    │
│    │     ├── ChromaDB vector search (RAG context)            │
│    │     └── LLM prompt → structured JSON                    │
│    └── Aggregate: severity, drowsiness, age warnings         │
│                                                              │
│  GET /api/v1/prescriptions  (history)                        │
│  GET /api/v1/health         (status)                         │
└──────────┬──────────────────────────┬───────────────────────┘
           │                          │
           ▼                          ▼
   ┌───────────────┐        ┌──────────────────┐
   │  LLM Provider │        │  ChromaDB        │
   │  (one of:)    │        │  (vector store   │
   │  • Gemini     │        │   for RAG)       │
   │  • OpenAI     │        └──────────────────┘
   │  • Ollama     │
   │  • HuggingFace│
   │  • Template   │
   └───────────────┘
```

### Two-phase frontend flow

The frontend splits the work into two sequential API calls so the user gets fast feedback:

**Phase 1 — Upload** (`POST /upload`): the image is sent to the server, OCR runs, and within a few seconds the user sees which medicines were detected and the OCR confidence score.

**Phase 2 — Analysis** (`POST /analysis`): using the prescription ID from phase 1, the backend analyses each medicine sequentially. The frontend animates through the medicine names to show progress. This phase can take 10–60 seconds depending on the number of medicines and the LLM provider speed.

### Cascading medicine lookup

For each medicine, the backend tries sources in order:

1. **ChromaDB** — local vector similarity search for RAG context (drug class, known info)
2. **LLM training knowledge** — the model's own clinical knowledge, enriched with RAG context
3. **YAML templates** — pre-written safety profiles by drug class (offline fallback)
4. **Error record** — if all else fails, an "analysis unavailable" placeholder is inserted

This means the system degrades gracefully: if ChromaDB is empty, the LLM still works. If the LLM is unavailable, templates serve a basic response.

### Storage

Uploaded images and prescription metadata (`.meta` JSON sidecar files) are stored on disk under `./uploads/`. PostgreSQL is optional — without it, all history data comes from these sidecar files. The frontend also caches results in `localStorage` for 24 hours for instant re-access.

│
└── working-backend/     # FastAPI backend
    ├── app/
    │   ├── api/routes/  # upload, analysis, prescriptions, health
    │   ├── providers/   # LLM provider implementations + factory
    │   ├── services/    # AnalysisService, LLMService, VectorStoreService

    │   ├── core/        # Settings, DB, rate limiter

    ├── data/            # medicine_seed.json for ChromaDB seeding

## Installation & Setup

### Prerequisites

These are required regardless of operating system:

- **Python 3.11+**
- **Node.js 20+** and **npm** (or pnpm / yarn)
- **Tesseract OCR** (system binary)
- **Git**
- A terminal / command prompt

PostgreSQL and Docker are optional.

---

### Windows

#### 1. Install Python

Download Python 3.11+ from [python.org](https://www.python.org/downloads/windows/). During installation, check **"Add Python to PATH"**.

Verify:
```cmd
python --version
pip --version
```

#### 2. Install Node.js

Download Node.js LTS from [nodejs.org](https://nodejs.org/). The installer includes npm.

Verify:
```cmd
node --version
npm --version
```

#### 3. Install Tesseract OCR

Download the Tesseract installer from [UB-Mannheim's GitHub releases](https://github.com/UB-Mannheim/tesseract/wiki). Install to `C:\Program Files\Tesseract-OCR\` (the default).

Verify:
```cmd
"C:\Program Files\Tesseract-OCR\tesseract.exe" --version
```

#### 4. Clone the repository

```cmd
git clone https://github.com/your-org/rxlens.git
cd rxlens
```

#### 5. Backend setup

```cmd
cd working-backend

python -m venv venv
venv\Scripts\activate

pip install -r requirements.txt
```

If you intend to use Gemini, install the extra package:
```cmd
pip install google-generativeai
```

Copy and configure the environment file:
```cmd
copy .env.example .env
```

Open `.env` in a text editor and set at minimum:
```env
MODEL_PROVIDER=gemini
GEMINI_API_KEY=your-key-here
TESSERACT_CMD=C:\Program Files\Tesseract-OCR\tesseract.exe
```

#### 6. Frontend setup

Open a new terminal window:
```cmd
cd rxlens

npm install
copy .env.local.example .env.local
```

`.env.local` only needs one variable and the default works for local development:
```env
NEXT_PUBLIC_API_URL=http://localhost:8000
```

## Environment Configuration

Both the frontend and backend have their own environment files. Here is a complete reference for all the variables you are likely to need to change.

### Backend — `.env`

#### Which LLM provider to use

Set `MODEL_PROVIDER` and the matching credential. Only set the credential for the provider you are using — leave the rest blank.

**Google Gemini (recommended — free tier available):**
```env
MODEL_PROVIDER=gemini
GEMINI_API_KEY=AIza...
MODEL_NAME=gemini-2.5-flash    # or gemini-1.5-pro, gemini-2.0-flash, etc.
```

**OpenAI:**
```env
MODEL_PROVIDER=openai
OPENAI_API_KEY=sk-...
MODEL_NAME=gpt-4o              # or gpt-4o-mini, gpt-3.5-turbo
```

**OpenAI-compatible API (e.g. Azure, Together.ai, Groq):**
```env
MODEL_PROVIDER=openai
OPENAI_API_KEY=your-key
OPENAI_BASE_URL=https://api.groq.com/openai/v1
MODEL_NAME=llama-3.3-70b-versatile
```

**Ollama (local, no API key):**
```env
MODEL_PROVIDER=ollama
OLLAMA_BASE_URL=http://localhost:11434
MODEL_NAME=llama3.2            # or gemma3, mistral, phi4, etc.
```
You must have Ollama installed and the model pulled: `ollama pull llama3.2`

**No LLM / offline mode:**
```env
MODEL_PROVIDER=template
```
Uses pre-written YAML templates. No internet or API key required. Useful for testing.

---

#### OCR configuration

```env
OCR_PROVIDER=tesseract         # or easyocr
TESSERACT_CMD=/usr/bin/tesseract   # adjust per OS (see install steps above)
TESSERACT_LANG=eng             # add languages: eng+hin, eng+tam, etc.
```

For EasyOCR (better on handwritten text, slower):
```bash
pip install easyocr
```
```env
OCR_PROVIDER=easyocr
```

---

#### Database (optional)

By default, prescription metadata is stored as `.meta` files alongside uploaded images — no database needed.

To enable PostgreSQL:
```env
DATABASE_URL=postgresql+asyncpg://user:password@localhost:5432/prescription_db
```

Create the database and run migrations:
```bash
createdb prescription_db
alembic upgrade head
```

---

#### Vector store / RAG (optional)

ChromaDB runs locally with no additional setup. To seed it with medicine data:
```bash
python scripts/seed_vector_store.py
```

To disable it entirely (LLM knowledge only):
```env
VECTOR_DB_ENABLED=false
```

---

#### Upload limits

```env
MAX_UPLOAD_SIZE_MB=10          # increase if you get 413 errors on large images
UPLOAD_DIR=./uploads           # change if you want uploads stored elsewhere
```

---

#### Production settings

```env
ENVIRONMENT=production         # hides internal error details from responses
SECRET_KEY=a-long-random-string-here   # generate with: python -c "import secrets; print(secrets.token_hex(32))"
CORS_ORIGINS=["https://your-frontend-domain.com"]   # lock down CORS
DEBUG=false
```

## Running the Project

You need two terminal windows — one for each service.

**Terminal 1 — Backend:**
```bash
cd working-backend
source venv/bin/activate     # Windows: venv\Scripts\activate
uvicorn app.main:app --reload --host 0.0.0.0 --port 8000
```

The backend is ready when you see `Startup complete.` in the logs. API docs are at `http://localhost:8000/docs`.

**Terminal 2 — Frontend:**
```bash
cd rxlens
npm run dev
```

Open `http://localhost:3000` in your browser.

## LLM Provider Quick-Start

Not sure which provider to use? Here is the fastest path for each:

| Goal | Provider | Steps |
|---|---|---|
| Try it immediately, no account | `template` | Just set `MODEL_PROVIDER=template` |
| Best quality, free tier | `gemini` | Get a key at [aistudio.google.com](https://aistudio.google.com), set `GEMINI_API_KEY` |
| OpenAI | `openai` | Get a key at [platform.openai.com](https://platform.openai.com), set `OPENAI_API_KEY` |
| Fully local, no internet | `ollama` | Install [Ollama](https://ollama.com), run `ollama pull llama3.2` |
| Local GPU | `huggingface` | `pip install transformers torch`, set `HF_MODEL_PATH` |

## Docker (Optional)

A `docker-compose.yml` is included in the backend directory. It starts the FastAPI app and a PostgreSQL instance together:

```bash
cd working-backend
docker compose up --build
```

The backend will be available at `http://localhost:8000`. You will still need to run the frontend separately with `npm run dev` unless you add it to the compose file.

Make sure your `.env` file is present in `working-backend/` before running Docker — Compose reads it automatically.
